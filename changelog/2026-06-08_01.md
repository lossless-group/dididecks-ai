## Why Care?

DidiDecks had the right repo shape but not yet a coherent product frontend. The active path in `dididecks-ai/` is no longer "keep thinking about the architecture"; it is now **build in `apps/frontend/` as SvelteKit and treat `deck-saas` as reference material only**.

That matters because three ambiguities closed in one pass:

1. **Frontend target ambiguity closed.** `apps/frontend/` is now the active product app. `apps/deck-shell/` and `splash/` remain references.
2. **Commercial surface gap closed.** The repo now has a real public route family for about, pricing, auth, billing results, contact, privacy, and terms.
3. **Migration discipline established.** `deck-saas` ideas were ported selectively into Svelte components and docs; React/Next runtime code was not pasted into the target app.

## What's New?

### Public/commercial routes now exist in the active SvelteKit app

The active frontend now includes:

- `/`
- `/about`
- `/pricing`
- `/billing/success`
- `/billing/cancel`
- `/contact`
- `/legal/privacy`
- `/legal/terms`
- `/auth/sign-in`
- `/auth/sign-up`

These are product surfaces only. Billing is intentionally not wired to secrets or fake checkout logic. Auth is intentionally a shell until backend session validation is connected.

### Workspace routes moved beyond placeholder state

The SvelteKit app now has navigable product pages for:

- `/dashboard`
- `/decks`
- `/decks/new`
- `/decks/[deckId]`
- `/decks/[deckId]/editor`
- `/decks/[deckId]/map`
- `/decks/[deckId]/smart-edit`
- `/decks/[deckId]/rebuild`
- `/decks/[deckId]/versions`
- `/decks/[deckId]/exports`
- `/decks/[deckId]/access`
- `/account/settings`

These are still mock-data-backed, but they now read as a product rather than an empty scaffold.

### Editor-side concepts from deck-saas were reimplemented as Svelte

`deck-saas/front-end/components/editor/` was treated as concept inventory, not code to transplant. The following Svelte components now exist in `apps/frontend/src/lib/components/editor/`:

- `CommentsPanel.svelte`
- `VersionsPanel.svelte`
- `ExportPanel.svelte`

They were integrated into `DeckEditorShell.svelte` alongside the existing:

- `DeckEditorShell.svelte`
- `SlideNavigator.svelte`
- `SlideCanvas.svelte`
- `BlockInspector.svelte`
- `AiCommandPanel.svelte`

### API boundaries are explicit

The frontend now documents and wraps:

- `POST /api/auth/session/validate`
- `GET /api/billing/plans`
- `GET /api/billing/me`
- `POST /api/billing/checkout`

using public-frontend wrappers only. No billing secrets, AI keys, or backend credentials were introduced.

### Repo docs now explain the migration

New root docs capture:

- public-pages and billing-surface scope
- billing frontend contract
- legal-draft status
- deck-saas to Svelte migration map
- component inventory
- frontend API contract

## How It Works

### `deck-saas` is now a source of ideas, not runtime authority

The useful source tree was `deck-saas/front-end/`, which is a Next.js 15 + React 19 app. That repo contains good product ideas:

- public route inventory
- dashboard layout patterns
- deck library patterns
- editor side panels
- billing/auth page structure
- API contract hints

But those ideas now land in `dididecks-ai/apps/frontend/` only after translation into SvelteKit-compatible route files, Svelte components, and local mock data contracts.

### Backend boundaries were preserved on purpose

The session explicitly did **not** do any of the following:

- add Stripe secrets
- add AI provider keys
- copy backend code into frontend
- fake checkout completion
- fake OAuth/session truth

The browser sees product surfaces. The backend remains the source of truth.

## Verification

Validated in `apps/frontend`:

- `pnpm --filter @dididecks/frontend check`
- `pnpm --filter @dididecks/frontend build`

Both passed after switching the API base URL lookup to a public env-safe SvelteKit/Vite pattern.

## What's Next

- Connect auth shell pages to the real backend session flow.
- Connect pricing and billing result pages to backend checkout/session confirmation.
- Replace local mock deck state with backend-backed CRUD, upload, smart-edit, rebuild, export, and access endpoints.
- Decide how much of the remaining `deck-saas` reference surface is still worth porting now versus leaving as reference until backend maturity catches up.
